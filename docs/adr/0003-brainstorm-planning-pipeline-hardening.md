# ADR 0003: Brainstorm Planning Pipeline Hardening

**Status**: Accepted
**Date**: 2026-04-07
**Epic**: aab2-356d — Strengthen brainstorm and preplanning: full scrutiny for split epics, feasibility-gap resolution, and LLM prompt alignment

---

## Context

The `/dso:brainstorm` → scrutiny pipeline → `/dso:preplanning` chain had three gaps that let planning-quality signals fall through unresolved:

1. **Follow-on epic scrutiny was skipped.** When brainstorm produced scope-splits (multiple follow-on epics), only the primary epic received the full scrutiny pipeline. Follow-on epics were created and handed to preplanning without gap analysis, web research, or scenario analysis. This caused follow-on epics to arrive at sprint with under-specified scope.

2. **Feasibility critical findings had no resolution loop.** The `dso:feasibility-reviewer` agent could emit critical findings, but the pipeline had no bounded re-entry path. Critical findings either surfaced at preplanning (too late) or were silently deferred. There was no structured mechanism for brainstorm to reattempt scope refinement after a feasibility gap was identified.

3. **LLM-instruction changes were not detected during brainstorm.** Epics that modified skill files, agent definitions, prompt templates, or hook behavioral logic required special handling — hallucination risk is higher, bot-psychologist review is valuable — but brainstorm had no signal to distinguish these epics from feature epics. The scrutiny pipeline ran identically regardless of whether the epic touched LLM-facing artifacts.

---

## Decision

### Follow-on epic scrutiny gate

Brainstorm Phase 3 Step 0 now invokes the scrutiny pipeline on each follow-on epic created by a scope-split. A depth cap of 1 prevents recursive scrutiny of epics generated by scrutiny itself. The `request_origin` field is set to `scope-split` to allow the scrutiny pipeline to adjust its focus. Part A (pre-strip) runs before the pipeline to normalize the epic.

The depth cap is not configurable — it is a hard invariant to prevent unbounded recursion.

### Feasibility-resolution gate

The scrutiny pipeline now uses `FEASIBILITY_GAP` annotation instead of spike recommendation for critical feasibility findings. This annotation is a structured signal — not free text — so it can be parsed reliably by the brainstorm orchestrator.

Preplanning emits `REPLAN_ESCALATE: brainstorm` when it encounters an unresolved `FEASIBILITY_GAP`. Brainstorm has a `FEASIBILITY_GAP` handler with a bounded re-entry loop governed by `brainstorm.max_feasibility_cycles` (default 2). If the gap is not resolved within the cycle limit, brainstorm escalates to the user.

### Prompt-alignment step (scrutiny pipeline Step 5)

The scrutiny pipeline gains a new terminal step — Step 5: Prompt Alignment. It fires only when the epic touches LLM-facing artifacts, detected via a canonical keyword list covering:

- Skill files (`plugins/dso/skills/**`)
- Agent definitions (`plugins/dso/agents/**`)
- Prompt templates (`plugins/dso/docs/prompts/**`, `plugins/dso/skills/shared/prompts/**`)
- Hook behavioral logic (`plugins/dso/hooks/**`)

Doc-only epics are excluded. When the keyword list matches, a `matched_keyword` state variable is set and `dso:bot-psychologist` is dispatched via the Agent tool to review the prompt-alignment implications of the epic's scope. When there is no match, Step 5 exits immediately without dispatch.

### Planning-intelligence log extensions

Three fields are added to the planning-intelligence log to make all three new mechanisms observable:

- `follow_on_scrutiny_depth` — depth at which follow-on scrutiny was invoked (0 = top-level, 1 = follow-on)
- `feasibility_cycle_count` and `feasibility_gap` — cycle counter and last gap text for the feasibility re-entry loop
- `llm_instruction_signal` and `matched_keyword` — whether the prompt-alignment step fired and which keyword matched

---

## Consequences

**Positive:**
- Follow-on epics arrive at sprint with the same scrutiny coverage as primary epics. Scope underspecification in follow-ons is caught at brainstorm, not at sprint mid-implementation.
- Feasibility gaps have a defined resolution path with a bounded cycle limit. Gaps no longer silently persist into preplanning or sprint.
- LLM-instruction epics receive bot-psychologist review during brainstorm, reducing hallucination risk in prompt and agent edits before implementation begins.
- All three mechanisms are observable via the planning-intelligence log without requiring skill-file inspection.

**Negative / Trade-offs:**
- Follow-on epics now incur a full scrutiny pipeline execution. For epics with many scope-splits, this increases brainstorm execution time proportionally.
- The feasibility re-entry loop adds latency when gaps exist. The default cycle limit of 2 may be insufficient for complex integration scenarios; increasing `brainstorm.max_feasibility_cycles` is possible but increases execution time.
- Bot-psychologist dispatch in Step 5 requires the Agent tool. When brainstorm runs as a sub-agent (Agent tool unavailable), Step 5 degrades to a warning log without dispatch — the scrutiny pipeline does not block.
- The depth cap of 1 for follow-on scrutiny is a hard invariant. If a follow-on epic itself produces scope-splits, those second-level follow-ons are not scrutinized.

**Config key introduced:**
- `brainstorm.max_feasibility_cycles` (default: 2) — maximum re-entry cycles in the feasibility-resolution gate before user escalation.

**Skills modified:**
- `plugins/dso/skills/brainstorm/SKILL.md`
- `plugins/dso/skills/preplanning/SKILL.md`
- `plugins/dso/skills/shared/workflows/epic-scrutiny-pipeline.md`
