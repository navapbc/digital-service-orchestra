# As a platform engineer, I can run project-detect.sh --suites to discover test suites

## Description

**What**: Add --suites flag to project-detect.sh that discovers test suites from Makefile targets, pytest directories, npm scripts, and bash test runners, outputting a JSON array with a defined schema.
**Why**: Core data contract enabling downstream CI workflow generation. Without structured discovery, every project onboarding requires manual test suite identification.
**Scope**:
- IN: --suites flag, JSON output (name, command, speed_class, runner), all four heuristic sources, name derivation rules, precedence ordering, dedup by name, backward compat, fixture-based acceptance test
- OUT: Config key declaration (separate story), CI workflow generation (Milestone B)

## Done Definitions

- When this story is complete, running project-detect.sh --suites on a repo with Makefile test targets outputs a JSON array with correct name/command/runner fields
  ← Satisfies: "project-detect.sh discovers individual test suites and outputs structured data"
- When this story is complete, running project-detect.sh (without --suites) produces identical output to the current baseline
  ← Satisfies: "backward compatibility constraint"
- When this story is complete, a fixture repo with Makefile test-unit/test-e2e and tests/models/ with test_*.py produces a JSON array with entries for make and pytest runners
  ← Satisfies: "fixture-based acceptance test"
- Unit tests written and passing for all new or modified logic

## Considerations

- [Testing] Multiple heuristic sources need fixture-based tests per SC4
- [Reliability] Precedence and dedup must handle overlapping heuristics robustly
- [Maintainability] Heuristic sources should be structured for future extension
