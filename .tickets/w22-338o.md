# Tech-stack-agnostic test gate


## Notes

**2026-03-21T04:32:02Z**


## Context
The DSO plugin's pre-commit test gate currently hardcodes Python conventions — it only checks .py source files and looks for test_<basename>.py. Host projects using other tech stacks (bash, TypeScript, Go) get no test gate coverage at all, as we demonstrated when a comment change to bump-version.sh passed the gate despite having an associated test-bump-version.sh. Since DSO is a portable plugin installed across projects with different stacks, the test gate needs to work without assuming any specific language or naming convention.

## Success Criteria
1. The test gate uses alphanum-normalized fuzzy matching: strip all non-alphanumeric characters from the staged source file's basename and from test filenames in configured test directories; a test file is 'associated' if its normalized name contains both the normalized source basename AND the string 'test'. Examples: bump-version.sh (normalized: bumpversionsh) matches test-bump-version.sh (normalized: testbumpversionsh); api.go matches api_test.go; version.py does NOT match test-bump-version.sh (normalized versionpy is not a substring of testbumpversionsh). The gate searches directories configured via test_gate.test_dirs in dso-config.conf (default: tests/)
2. A .test-index file at the repo root (checked into version control, manually maintained or generated by /dso:dev-onboarding) maps source files to tests that don't follow naming conventions. Format: one entry per line, source/path.ext: test/path1.ext, test/path2.ext. When a mapped source file is staged, those tests are added to the required set alongside any fuzzy matches. Colons and commas in paths are not supported
3. During pre-commit, any .test-index entry whose test file does not exist on disk is removed from the index and the modified index is staged automatically (the hook modifies the file on disk and stages it, similar to how format hooks modify and re-stage files). If all test paths for a source entry are removed, the entire line is removed
4. /dso:dev-onboarding includes a step that scans the project for source-to-test associations the fuzzy match would miss, generates a .test-index file with those mappings, and reports the coverage summary (files with fuzzy matches, files with index entries, files with no test coverage). This is dogfooded against this repo to produce its initial .test-index and validate the fuzzy match algorithm against real naming conventions
5. After deploying to this repo (dogfooding), staging bump-version.sh triggers the test gate to require test-bump-version.sh — the same scenario that previously passed unchecked. A test case reproduces the original gap and verifies it is now caught
6. The test gate completes within the existing pre-commit timeout: a benchmark test stages 20 files with associated tests and verifies the gate returns within 10 seconds
7. /dso:implementation-plan guidance is updated to recommend test filenames that follow fuzzy-matchable conventions, and to note that unconventional names require a .test-index entry

## Dependencies
None — this replaces the existing test gate in-place. The existing two-layer gate architecture (pre-commit hook + PreToolUse bypass sentinel) is preserved; only the test association logic within pre-commit-test-gate.sh changes.

## Approach
Three-layer system: (1) zero-config fuzzy matching for standard conventions, (2) optional .test-index for explicit associations, (3) upstream enforcement via /dso:implementation-plan guidance. /dso:dev-onboarding bootstraps the .test-index for existing projects. Self-healing: stale index entries are auto-pruned during commit.
