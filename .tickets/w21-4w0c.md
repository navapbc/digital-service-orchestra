# Test gate blocks commits when RED tests exist for unimplemented functions

## Description
The pre-commit test gate (pre-commit-test-gate.sh) blocks commits when RED tests exist in a file associated with a changed source file. This breaks TDD workflow:

1. Developer writes RED tests for functions that don't exist yet (e.g., map_status, process_inbound)
2. Developer implements a different function in the same source file (e.g., pull_comments in bridge-inbound.py)
3. Test gate discovers test_bridge_inbound.py via fuzzy match on bridge-inbound.py
4. Test gate runs ALL tests in the file — including the RED tests for unimplemented functions
5. RED tests fail (AttributeError: module has no attribute) → commit blocked

## Current Workaround
Stash RED test file changes before committing implementation, then pop stash after. This is fragile and error-prone.

## Root Cause
RED tests have no annotation distinguishing them from GREEN tests. The test gate cannot tell that a test is intentionally failing because the implementation hasn't been written yet.

## Proposed Fix Options
1. **Convention**: RED test tasks annotate tests with `@pytest.mark.xfail(reason='RED: awaiting implementation')` — test gate skips xfail-marked tests
2. **File separation**: RED tests go in separate files (e.g., test_bridge_inbound_red.py) that are excluded from the test gate until implementation commits
3. **Test gate awareness**: test gate checks if failing tests are AttributeError/ImportError on missing functions and treats those as RED (not blocking)
4. **.test-index exemption**: Add a `.test-index` entry that maps source files to only the GREEN test subset

## Impact
Every TDD sprint batch that creates RED tests before implementation hits this issue. Observed in w21-bwfw sprint batches 7-9.
