# Ticket system v3 — migration from tk and cutover

## Context
After the new ticket system is feature-complete (dso-0k2k, w21-54wx, w21-bwfw), all existing tickets must be migrated and every reference to tk across skills, scripts, hooks, and documentation must be updated. This is the highest-blast-radius epic in the series — the migration session modifies the very tools it uses to track its own work, and a failed cutover could leave the plugin in a state where neither system works. The cutover must be orchestrated by a self-healing script that saves detailed output, reverts to a known-good state on any failure, and can be dry-run on a branch before executing on main. No other sessions should be active during cutover.

## Success Criteria
1. A cutover script orchestrates 5 phases (pre-flight, data migration, reference update, validation, cleanup) with verification gates between each phase. Each phase logs detailed output to a timestamped log file. On any failure, the script reverts to the last known-good state and exits with a clear error and log path. The script is idempotent — re-running after a failure resumes safely.
2. Pre-flight captures a comprehensive snapshot: full tk show output for every ticket, ticket counts, dependency graphs, and Jira mappings. This snapshot is used for semantic field-level comparison in Phase 4 (not text-diff formatting comparison).
3. Data migration uses the new ticket CLI commands (not raw event writes) to convert .tickets/*.md into events, preserving ticket IDs (old format IDs accepted by the new system), status, dependencies, parent/child relationships, notes with timestamps, and Jira key mappings. Migration is idempotent — skips already-migrated tickets on re-run. Malformed tickets logged and skipped. Compaction disabled during migration.
4. Reference update is a single atomic commit that updates all tk references using enumerated command-specific patterns (not blanket substring replacement). Includes documentation updates deferred from Epics 1-3 (CLAUDE.md architecture, quick reference, path references). After update, a comprehensive grep verifies no remaining tk references exist. Syntax validation (bash -n, py_compile) gates the commit.
5. After cutover, validate.sh --ci passes, ticket list/show/deps produce correct results compared against the pre-flight snapshot (semantic field comparison), and /dso:sprint on the project backlog completes workflows using only the new system. Jira bridge validation is optional (external dependency doesn't block local validation).
6. Cleanup (removal of .tickets/, tk script, tk-specific test fixtures) is a separate commit, executed only after explicit confirmation. A git tag (pre-cleanup-migration) preserves access to old data. Either commit (reference update or cleanup) can be independently reverted via git revert.
7. The cutover script supports dry-run on a separate branch, producing the same log output and verification checks without modifying main.
8. A complete CLI command reference documents all ticket commands (init, create, show, list, transition, comment, link, unlink, deps, sync, archive, bridge-status, bridge-fsck, and --format=llm output mode) with usage examples, options, and expected output. This is the authoritative documentation for the new ticket system.

## Dependencies
w21-bwfw (Jira bridge — system must be feature-complete before cutover)

## Cross-Epic Dependencies
- dso-0k2k must support old ticket ID formats (e.g., dso-0k2k, w21-54wx) for migrated tickets — the new system's ID validation cannot be restricted to TKT-xxxxxxxx format only

## Approach
Self-healing cutover script with 5 phased gates. Migration uses new ticket commands to self-test the system. Reference update uses enumerated patterns with syntax validation gate. Semantic comparison against pre-flight snapshot for validation. Pre-cleanup git tag preserves old data. Supports dry-run. All documentation (CLAUDE.md, CLI reference, architecture docs) updated as part of the atomic reference update commit.

## Migration Failure Modes (from scenario analysis)
- Malformed frontmatter: logged, skipped, operator fixes before retry
- Special characters in notes: all text through Python json.dumps (Epic 1 convention)
- Dependencies referencing non-existent tickets: logged as warnings, LINK events still created
- Compaction during migration: disabled (threshold set to max)
- Old ticket IDs: preserved (new system must accept old format)
- Partial migration crash: idempotent re-run skips completed tickets
- Reference update precision: enumerated patterns, post-update grep, syntax validation gate
- Pre-cleanup git tag: old data recoverable via git show
- Jira bridge validation: optional (external dep doesn't block local validation)

## Success Criterion #9 — Commit Ticket Gate

A commit-msg stage pre-commit hook (pre-commit-ticket-gate.sh) blocks commits that don't reference at least one valid ticket ID in the commit message. Valid means the ID matches the v3 format (XXXX-XXXX hex) and the ticket exists in the v3 system (directory + CREATE event check). The hook loads review-gate-allowlist.conf and skips enforcement when all staged files match the allowlist. Merge commits (MERGE_HEAD present) are exempt. The hook's error output includes the expected ID format and a pointer to creating a ticket. The hook is included in pre-commit-config.example.yaml and registered by dso-setup.sh / /dso:project-setup during host project onboarding. Validation: after one sprint of dogfooding, a retrospective git-log query confirms all gated commits reference real tickets and no false positives occurred on allowlisted-only changesets.

Context: This criterion is part of the migration cutover — the gate enforces v3 ticket references, completing the transition from the old system by ensuring all future reviewable commits are traceable through v3. Without this enforcement, the migration is technically complete but practically unenforced. SC#9 consumes but does not modify the shared gate infrastructure (review-gate-allowlist.conf, pre-commit-wrapper.sh, Layer 2 bypass sentinel).

Approach: commit-msg hook following the same pattern as the review gate and test gate — loads the shared allowlist, uses the pre-commit-wrapper.sh timeout mechanism, and gets Layer 2 bypass protection from the existing sentinel.

New dependency: dso-0k2k (v3 storage layer must be stable and queryable for the existence check).
