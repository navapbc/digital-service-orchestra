# Ticket system v3 — dependency management, cross-worktree sync, and conflict resolution

## Context
Once the core storage engine exists (dso-0k2k), two capabilities are needed before the system can support multi-agent orchestration: dependency tracking (so agents know which tickets are unblocked) and cross-environment sync (so multiple sessions see consistent state). Without dependency management, agents cannot autonomously pick the next task. Without sync, sessions diverge. Without conflict resolution, the first sync after divergence produces arbitrary state that may not reflect the most informed session's work.

## Success Criteria
1. `ticket link` creates LINK events (blocks, depends_on, relates_to). `ticket unlink` removes relations. Cycle detection prevents circular dependencies.
2. `ticket deps <id>` returns dependency graph with `ready_to_work` boolean — true when all direct blockers are closed. Graph traversal completes in under 2 seconds with 1,000 tickets.
3. When a ticket is transitioned to closed, the system identifies newly unblocked tickets in the output. After `ticket sync` resolves remote status changes, the same unblock check runs and reports newly unblocked tickets in sync output.
4. Cross-worktree visibility via shared `.tickets-tracker/` symlinks. All commands resolve to canonical real path before flock to prevent symlink/real-path locking inconsistency.
5. `ticket sync` uses a split-phase design to avoid holding flock during network I/O: (a) `git fetch` without flock, (b) acquire flock, (c) `git rebase` + conflict resolution + unblock check (local only, <2s), (d) release flock, (e) `git push` without flock with retry on non-fast-forward. Network phases have 30-second individual timeouts.
6. After sync, when multiple environment IDs produced STATUS events for the same ticket, the environment with more STATUS events (not total events) has its latest STATUS treated as authoritative, with timestamp tiebreaker. All events preserved. Conflict log records every resolution.
7. Dependency-aware archiving: closed tickets with no open dependents or children can be archived. Archived tickets retain a lightweight tombstone (ID, type, final status) for dependency resolution. Compaction must sync before compacting (pull remote events first) and skip tickets that already have a remote SNAPSHOT to prevent concurrent-compaction git conflicts.
8. A PreToolUse hook prevents agents from manually editing event files in `.tickets-tracker/` — blocks Edit, Write, and Bash modifications with a message redirecting to `ticket` commands.
9. No single sync or dependency operation exceeds 60 seconds total wall time. flock is held only during local git operations (<2s), never during network I/O.

## Dependencies
dso-0k2k (core storage engine)

## Approach
LINK events for dependency tracking. Graph traversal via python3 reducer. Symlink-based worktree sharing with canonical path resolution. Split-phase sync (fetch unlocked, rebase locked, push unlocked). Most-STATUS-events-wins conflict resolution with environment ID. PreToolUse hook for agent guardrail.

## Red Team / Blue Team Review
Architecture reviewed via adversarial red-team (18 findings) and independent blue-team validation. Key findings incorporated: shared flock for sync (split-phase design), STATUS-only event counting, auto-unblock after sync, path canonicalization, compaction-sync ordering. Additional scenario analysis identified: concurrent compaction git conflict (fix: sync-before-compact + skip remote SNAPSHOTs), SNAPSHOT compacted_at timestamp (added to Epic 1). Split-phase sync scenarios validated: agent writes during fetch (safe), writes between flock release and push (safe — pushed on next sync), concurrent sync on same machine (safe — serialized by flock), push race (safe — retry loop).

## Cross-Epic Note (from w21-bwfw red/blue team review)

Bridge-authored STATUS events (from the Jira bridge's dedicated environment ID) must be EXCLUDED from the most-STATUS-events-wins conflict resolution count. The bridge is a relay, not an independent decision-maker — its events should not accumulate authority over local environments. Without this exclusion, the bridge becomes permanently authoritative for every synced ticket after enough sync cycles, undermining the conflict resolution heuristic.
