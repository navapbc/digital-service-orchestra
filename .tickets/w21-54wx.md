# Ticket system v3 — dependency management and cross-worktree sync


## Notes

**2026-03-20T03:28:47Z**

## Context
Once the core storage engine exists (dso-0k2k), two capabilities are needed before the system can support multi-agent orchestration: dependency tracking (so agents know which tickets are unblocked and ready to work) and cross-worktree sync (so multiple sessions on the same machine or across environments see consistent ticket state). Without dependency management, agents cannot autonomously pick the next task. Without sync, sessions operate in isolation and diverge.

## Success Criteria
1. ticket link <source> <relation> <target> creates LINK events supporting blocks, depends_on, and relates_to relations
2. ticket deps <id> returns the dependency graph for a ticket including a ready_to_work boolean — true only when all blockers are closed
3. When a ticket is transitioned to closed, the system automatically identifies newly unblocked tickets and includes them in the command output
4. Cross-worktree visibility: all worktrees on the same machine see the same ticket state without manual sync, via shared .tickets-tracker/ symlinks
5. ticket sync pushes local events and pulls remote events from the tickets branch — concurrent pushes from different environments resolve without conflict due to append-only event structure
6. Dependency graph traversal completes in under 2 seconds with 1,000 tickets
7. Dependency-aware archiving: closed tickets with no open dependents can be archived (removed from active set). Tickets with open dependents are protected from archiving

## Dependencies
dso-0k2k (core storage engine)

## Approach
LINK events for dependency tracking. Graph traversal via the python3 reducer. Symlink-based worktree sharing. Git push/pull for remote sync.
