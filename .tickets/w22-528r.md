# Setup skill: auto-discover host project test suites for CI workflow generation


## Notes

**2026-03-21T16:28:31Z**

## Problem

When /dso:project-setup helps a new project create a CI process from our example (examples/ci.example.yml), it copies a hardcoded template with specific job names (test-unit-agents, test-unit-services, etc.) that only match one project's structure. The host project's actual test suites are not incorporated — they have to be added manually.

## Current Behavior

- For projects with no CI: copies ci.example.yml as-is (static template)
- For projects with existing CI: runs guard analysis (test/lint/format present?) but doesn't modify the workflow
- project-detect.sh detects Makefile targets and test/lint/format presence, but does NOT discover individual test suites
- Config keys (commands.test_unit, commands.test) are generic — they don't map to per-suite CI jobs

## Investigation Findings

Key files:
- plugins/dso/scripts/dso-setup.sh (lines 353-440): CI workflow creation logic
- plugins/dso/scripts/project-detect.sh (lines 216-260): CI workflow detection
- plugins/dso/skills/project-setup/SKILL.md: orchestrates setup including CI
- examples/ci.example.yml: static template copied for new projects

project-detect.sh already extracts Makefile targets (grep for top-level targets) and checks for test/lint/format commands in existing workflows. But it stops at boolean presence detection — no discovery of individual test directories or suite structure.

## Desired Behavior

1. Extend project-detect.sh to discover test suite directories and their associated Makefile targets (e.g., tests/unit/agents/ -> make test-unit-agents, tests/hooks/ -> bash tests/run-hooks.sh)
2. When generating the CI workflow for a new project, dynamically template job definitions from detected suites instead of copying a static file
3. For existing CI workflows missing test guards, offer to add jobs for discovered-but-uncovered test suites

**2026-03-21T16:50:58Z**

## Refined Spec (replaces placeholder)

## Context
A platform engineer onboarding a new project to DSO spends ~30 minutes manually identifying test suites and editing CI configuration, because project-detect.sh only reports boolean test/lint/format presence — it doesn't discover individual test suites or their characteristics. This milestone adds a --suites flag that outputs structured JSON discovery. A platform engineer can immediately run project-detect.sh --suites to inspect what DSO discovered about their project's test structure, even before downstream CI generation (Milestone B) ships.

## Success Criteria
1. project-detect.sh --suites [REPO_ROOT] outputs a JSON array to stdout (empty array if none found, exit 0 always). Each element has keys: name (string, unique), command (string), speed_class (enum: fast|slow|unknown), runner (enum: make|pytest|npm|bash|config). Without --suites, existing KEY=VALUE output is unchanged (backward compatible). Auto-discovered suites default to speed_class=unknown; only config-declared suites may set fast or slow. Warnings go to stderr only.
2. Discovery heuristics applied in precedence order: config keys > Makefile targets matching /^test[-_]/ > tests/ or test/ directories containing test_*.py > package.json scripts starting with test > executable files matching test-*.sh or run-tests*.sh. When multiple heuristics match the same command, one entry emitted using highest-precedence runner. name is the deduplication key.
3. Unknown testing patterns declared in .claude/dso-config.conf using test.suite.<name>.command=<shell command> and test.suite.<name>.speed_class=fast|slow. Config entries merge with auto-discovered suites by name: config wins on all fields it explicitly sets; auto-discovered fields fill gaps. Config-only suites (no auto-discovered match) have runner=config.
4. Fixture-based acceptance test: given a repo with Makefile test-unit and test-e2e targets, a tests/ directory with test_*.py files, and a config-declared suite, project-detect.sh --suites outputs a JSON array with entries for all three runner types (make, pytest, config) with correct fields.

## Approach
Detection-driven template engine (Option A from brainstorm). Extend project-detect.sh with --suites flag. Additive change — existing output unchanged.

## Dependencies
Backward compatibility constraint: existing project-detect.sh KEY=VALUE output must not change. Milestone B (CI workflow generation) depends on this.
