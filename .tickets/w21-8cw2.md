# As a developer, local ticket changes are automatically pushed to Jira via the outbound bridge


## Notes

**2026-03-20T15:46:38Z**

## Description
**What**: GitHub Actions workflow (push trigger) that processes new local events, creates/updates Jira issues via ACLI, and writes SYNC events mapping local IDs to Jira keys.
**Why**: Walking skeleton — proves the bridge pipeline end-to-end.
**Scope**:
- IN: GitHub Actions workflow, push trigger, concurrency group, bridge git identity + commit-author skip, git diff for new events, ACLI create/update, verify-after-create, SYNC events, compiled-state outbound for STATUS, echo prevention (check existing SYNC before push)
- OUT: Inbound pull (w21-gykt), comment sync (w21-dww7), hardening (w21-2r0x)

## Done Definitions
- GitHub Actions workflow triggers on tickets branch push, processes new events via git diff, and creates/updates Jira issues ← Satisfies SC1
- Concurrency group ensures one bridge instance at a time; bridge commits use dedicated git identity and workflow skips its own commits ← Satisfies SC1
- SYNC events map local ID to Jira key after verified creation ← Satisfies SC4
- Outbound STATUS pushes compiled state (post-conflict-resolution), not raw events ← Satisfies SC4
- Before creating a Jira issue, bridge checks for existing SYNC event to prevent echo of inbound-imported tickets ← adversarial review
- Unit tests passing

## Considerations
- [Security] Jira API credentials in GitHub Actions secrets — no token logging in workflow output
- [Reliability] ACLI network failures need retry with backoff
- [Maintainability] Status/type mapping config must be version-controlled
