# Integrate TDD into implementation plan skill

TDD principles should be incorporated into the implementation plan skill. Stories that change code should start by writing a RED unit test that fails, and require tests to pass as one of the acceptance criteria for the task that completes the relevant code changes. Writing RED tests must be a separate task that is a dependency of writing implementation code. This should not be the only acceptance criteria. All tasks created by implementation plan should include acceptance criteria following the same logic the sprint skill uses to add acceptance criteria to tasks that are missing them. Testing should pass hypothesis assertion density standards and code coverage standards. When implementing this epic, we need to investigate patterns used to identify code that would be unreasonable to cover with unit tests. We want to provide an escape hatch for stories where adding unit tests would not be follow testing best practices, but we need to safeguard against agents using this escape hatch to avoid writing unit tests that would be required by testing best practices.

## Context
Implementation plans generated by /dso:implementation-plan do not enforce TDD discipline — tasks that change code are created without a preceding RED test task, and acceptance criteria do not require tests to pass. Agents producing these plans have no structural forcing function toward test-first development, and no principled escape hatch exists to distinguish genuinely untestable code from code that agents are avoiding testing. The result is implementation work that skips unit tests or buries them as afterthoughts.

## Success Criteria
- Every task that changes code with behavioral content (conditional logic, data transformation, decision points) has a preceding RED test task as a declared dependency
- Tasks classified as test-exempt include a stated justification referencing one of three criteria: (1) no conditional logic — pure wiring/assignment/initialization; (2) change-detector test — assertion would mirror implementation; (3) infrastructure-boundary-only code
- Exemption justifications are validated by a sub-agent reviewer during plan review — tasks whose justification doesn't meet the criteria are revised before the plan is presented to the user
- All tasks generated by the skill include acceptance criteria that require tests to pass (using the configured TEST_CMD)
- The test-exempt criteria are documented in the skill such that an agent given an ambiguous case (e.g., updating CLAUDE.md) would correctly classify it as exempt or not based on behavioral content, not file type

## Dependencies
None (independent of dso-s991)

## Approach
Strengthen the implementation-plan skill to: (1) always create a RED test task as a dependency before any behavioral-code task; (2) require test-exempt tasks to state a justification against the three behavioral-content criteria; (3) validate exemption justifications via sub-agent review during plan review before presenting to the user.

## Spec Amendment: Integration Tests

Tasks that touch external integration boundaries (external API calls, DB writes, file I/O as primary purpose) include a corresponding integration test task. Unlike unit test tasks, integration test tasks do not require a RED-first dependency — they may be ordered alongside or after the implementation task.

Integration test tasks follow the same escape hatch pattern as unit test tasks: a task may be marked integration-test-exempt only when justified against one of two criteria: (1) the boundary is already covered by existing integration tests that require no changes; (2) no automated test environment exists for the integration. Justifications are validated by the same plan reviewer sub-agent that validates unit test exemptions.

Additional success criteria:
- Tasks touching external integration boundaries include an integration test task (no RED-first dependency required)
- Integration test task exemptions require a stated justification referencing one of two criteria (existing coverage / no test environment), validated by the plan reviewer sub-agent
