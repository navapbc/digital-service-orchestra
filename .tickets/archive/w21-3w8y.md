# Preplanning adds epic's own children as deps, making the epic self-blocked


## Description

When `/dso:preplanning` decomposes an epic into child stories, it adds those children to the epic's `deps` (blockers) list. This causes the epic to appear as BLOCKED in `sprint-list-epics.sh` output, since its own children are open and listed as dependencies.

Example: `dso-ppwp` has 4 children (w22-uqfn, w22-sulb, w22-8jaf, w22-4ujv) which are all listed in its `deps` field. The epic shows as BLOCKED even though it is ready to sprint on — the children ARE the work, not external blockers.

## Expected Behavior

An epic's children should be listed in the `children` relationship, not in `deps`. The epic should remain unblocked and selectable in `/dso:sprint` when its only relationships are parent-child.

## Actual Behavior

Children are added to `deps`, causing the epic to self-block. `sprint-list-epics.sh` reports it as BLOCKED.

## Reproduction

1. Run `/dso:preplanning` on any epic
2. Run `tk show <epic-id>` — observe `deps` contains child IDs
3. Run `sprint-list-epics.sh --all` — epic appears in BLOCKED section

## File Impact

- `plugins/dso/skills/preplanning/` — likely where `tk dep` is called to wire children as blockers
- `plugins/dso/scripts/sprint-list-epics.sh` — could also filter out self-child deps when computing blocked status
