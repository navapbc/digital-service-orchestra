# As a developer, the Jira bridge handles failures gracefully without corrupting ticket state


## Notes

**2026-03-20T15:47:13Z**

## Description
**What**: Bridge hardening: flap detection, destructive change guards, relationship rejection persistence, incremental checkpointing, auth failure fast-abort, dual-timestamp checkpoint.
**Why**: Without hardening, the bridge can corrupt ticket state through oscillation, destructive overwrites, or partial failures.
**Scope**:
- IN: Flap detection (N oscillations → BRIDGE_ALERT + halt STATUS push), destructive change guards (empty desc, relationship removal, type downgrade, whitespace-as-empty), relationship rejection persistence (jira_sync_status: rejected), incremental checkpointing per batch, dual-timestamp checkpoint (last_pull_timestamp advances only on full-run success), fast-abort on auth failure (401)
- OUT: Observability UI (w21-qjcy), REVERT mechanism (w21-qjcy)

## Done Definitions
- Flap detection halts STATUS push for a ticket after N oscillations and emits BRIDGE_ALERT ← Satisfies SC6 hardening
- Destructive change guards reject empty-over-non-empty (whitespace treated as empty), relationship removal, and type downgrade ← Satisfies SC6
- Jira relationship rejections persist locally with jira_sync_status: rejected — local relationship never removed ← Satisfies SC6
- Checkpoint advances last_pull_timestamp only after full run success; per-batch checkpointing is resume-only ← Satisfies SC3 + adversarial review
- Bridge fast-aborts on auth failure (401), preserving checkpoint ← Satisfies SC6
- Unit tests passing

## Considerations
- [Reliability] Flap detection requires both outbound (S1) AND inbound (S2) STATUS events to detect bidirectional oscillation
- [Reliability] Dual-timestamp checkpoint must integrate with both git-diff (outbound) and JQL-windowed (inbound) models
