# As a developer, the cutover script orchestrates migration phases with logging, rollback, and dry-run


## Notes

**2026-03-20T17:24:14Z**

## Description
**What**: Build the cutover script skeleton: 5 phase gates (pre-flight, migrate, validate, reference update, cleanup), timestamped logging, rollback on failure, dry-run mode, idempotent re-run.
**Why**: The orchestration harness must be reliable before any migration logic runs. A broken script leaves the system in an unrecoverable state.
**Scope**:
- IN: Phase gate framework, logging to timestamped file, rollback (git checkout HEAD for uncommitted, git revert for committed), dry-run mode (same output without modifying main), idempotent resume from last successful phase
- OUT: Migration logic (w21-7mlx), reference update logic (w21-wbqz), validation logic (w21-25mq)

## Done Definitions
- Cutover script executes 5 phases sequentially with verification gates ← Satisfies SC1
- On any failure, script reverts to last known-good state and exits with error + log path ← Satisfies SC1
- Script supports dry-run on a separate branch ← Satisfies SC7
- Script is idempotent — re-running resumes from last successful phase ← Satisfies SC1
- Unit tests passing

## Considerations
- [Reliability] Rollback is a linear stack (revert most recent committed phase). Script executes phases sequentially, not as a DAG.
- [Reliability] Script does NOT use tk for its own tracking — standalone bash with file-based logging
